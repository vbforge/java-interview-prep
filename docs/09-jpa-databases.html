Q78
Hibernate fetch types — EAGER and LAZY

Short answer

EAGER fetching loads the associated entity immediately (in the same query). LAZY fetching loads the associated entity only when accessed (on-demand). LAZY is the recommended default for performance.

In depth

FeatureEAGERLAZY
Load timingImmediately (same query)On-demand (when accessed)
PerformancePotentially slower (more data)Faster initial load
N+1 riskLowerHigher (if not careful)
Default for@ManyToOne, @OneToOne@OneToMany, @ManyToMany
RecommendedOnly when necessary✅ Default preference
 
@Entity
public class User {
    @Id
    @GeneratedValue
    private Long id;
    
    // LAZY fetch (default for OneToMany)
    @OneToMany(mappedBy = "user", fetch = FetchType.LAZY)
    private List<Order> orders;
    
    // EAGER fetch (default for ManyToOne)
    @ManyToOne(fetch = FetchType.EAGER)
    private Role role;
    
    // Override to LAZY
    @ManyToOne(fetch = FetchType.LAZY)
    private Department department;
}

// LAZY access - requires open Session
User user = userRepository.findById(1L).orElseThrow();
// user.getOrders() returns proxy, not loaded yet
List<Order> orders = user.getOrders();  // Now loads from DB

LAZY initialization exceptions:

  • LazyInitializationException: Occurs when accessing LAZY property outside of transaction
  • Solution 1: Keep session open until data is accessed
  • Solution 2: Use @Transactional to keep session open
  • Solution 3: Use JOIN FETCH in JPQL
  • Solution 4: Use EntityGraph to define loading strategy
// Fix: Use JOIN FETCH to load eagerly in query
@Query("SELECT u FROM User u JOIN FETCH u.orders WHERE u.id = :id")
User findByIdWithOrders(@Param("id") Long id);
// JUNIOR NOTE: Always prefer LAZY for associations unless you have a specific reason for EAGER. EAGER loading can cause performance issues and N+1 problems. If you need data, load it explicitly with JOIN FETCH or EntityGraph.
Q79
@OneToOne, @OneToMany, @ManyToMany

Short answer

These JPA annotations define entity relationships: @OneToOne (single-to-single), @OneToMany (one-to-multiple), @ManyToOne (multiple-to-one), and @ManyToMany (multiple-to-multiple). The owning side has the foreign key.

In depth

// @OneToOne - bidirectional with shared primary key
@Entity
public class User {
    @Id
    private Long id;
    
    @OneToOne(cascade = CascadeType.ALL)
    @JoinColumn(name = "profile_id")  // Owning side
    private Profile profile;
}

@Entity
public class Profile {
    @Id
    private Long id;
    
    @OneToOne(mappedBy = "profile")  // Inverse side
    private User user;
}

// @OneToMany / @ManyToOne - bidirectional
@Entity
public class Order {
    @Id
    @GeneratedValue
    private Long id;
    
    @ManyToOne  // Owning side (has foreign key)
    @JoinColumn(name = "user_id")
    private User user;
}

@Entity
public class User {
    @OneToMany(mappedBy = "user")  // Inverse side
    private List<Order> orders;
}

// @ManyToMany - requires join table
@Entity
public class Student {
    @Id
    @GeneratedValue
    private Long id;
    
    @ManyToMany
    @JoinTable(
        name = "student_course",
        joinColumns = @JoinColumn(name = "student_id"),
        inverseJoinColumns = @JoinColumn(name = "course_id")
    )
    private Set<Course> courses;
}

@Entity
public class Course {
    @Id
    @GeneratedValue
    private Long id;
    
    @ManyToMany(mappedBy = "courses")  // Inverse side
    private Set<Student> students;
}

Relationship summary:

AnnotationCardinalityForeign Key SideDefault Fetch
@OneToOne1 ↔ 1Owning sideEAGER
@OneToMany1 ↔ NMany side (@ManyToOne)LAZY
@ManyToOneN ↔ 1Many sideEAGER
@ManyToManyN ↔ NJoin tableLAZY
// JUNIOR NOTE: The owning side has the foreign key. For bidirectional relationships, one side must be the owner (with @JoinColumn) and the other side is the inverse (with mappedBy). Only the owning side's changes are persisted automatically.
Q80
N+1 problem

Short answer

The N+1 problem occurs when you fetch a list of entities (1 query) and then access a lazy association on each entity (N additional queries). This results in N+1 database queries instead of a single optimized query.

In depth

// ❌ N+1 problem example
// N+1 = 1 query for users + N queries for orders

// Query 1: Get all users
List<User> users = userRepository.findAll();  // 1 query

// Query N+1: For each user, get their orders
for (User user : users) {
    List<Order> orders = user.getOrders();  // N queries!
    System.out.println(user.getName() + ": " + orders.size());
}

// Total queries: 1 + N
// If 1000 users → 1001 database queries!

Solutions to N+1:

// ✅ Solution 1: JOIN FETCH in JPQL
@Query("SELECT DISTINCT u FROM User u JOIN FETCH u.orders")
List<User> findAllWithOrders();

// ✅ Solution 2: EntityGraph
@EntityGraph(attributePaths = {"orders"})
@Query("SELECT u FROM User u")
List<User> findAllWithOrders();

// ✅ Solution 3: @NamedEntityGraph
@NamedEntityGraph(
    name = "User.orders",
    attributeNodes = @NamedAttributeNode("orders")
)
@Entity
public class User {
    // ...
}

// Usage with repository
@EntityGraph("User.orders")
List<User> findAll();

// ✅ Solution 4: EAGER fetch (not recommended)
@OneToMany(fetch = FetchType.EAGER)
private List<Order> orders;  // Avoid! Causes other issues

Detecting N+1:

  • Enable SQL logging: spring.jpa.show-sql=true
  • Use Hibernate statistics: spring.jpa.properties.hibernate.generate_statistics=true
  • Use tools: Hibernate Query Statistics, datasource-proxy
  • Look for: Many similar queries executed in loops
// JUNIOR NOTE: The N+1 problem is one of the most common performance issues in JPA applications. Always use JOIN FETCH or EntityGraph for collections you know you'll access. Don't rely on EAGER fetching — it can cause performance issues elsewhere.
Q81
What is optimistic locking? How is it implemented in databases?

Short answer

Optimistic locking assumes conflicts are rare. It uses a version number (or timestamp) that increments on each update. When updating, the database checks if the version matches the value read — if not, it throws an OptimisticLockException.

In depth

// Optimistic locking with @Version
@Entity
public class Product {
    @Id
    @GeneratedValue
    private Long id;
    
    private String name;
    private Integer quantity;
    
    @Version
    private Integer version;  // Automatically managed
}

// How it works:
// 1. Read product: version = 1
// 2. Update product: WHERE id = ? AND version = 1
// 3. If version is still 1 → update succeeds, version becomes 2
// 4. If version changed → update fails, OptimisticLockException

// SQL generated:
// UPDATE product SET name=?, quantity=?, version=2
// WHERE id=? AND version=1

Optimistic locking mechanisms:

MechanismImplementation
Version number@Version field (integer or long)
Timestamp@Version field (Instant or Timestamp)
Manual checkExplicit version check in query
 
// Handling optimistic lock exception
try {
    productRepository.save(product);
} catch (OptimisticLockException e) {
    // Refresh product and retry with new version
    Product freshProduct = productRepository.findById(product.getId()).orElseThrow();
    freshProduct.setQuantity(product.getQuantity() - 1);
    productRepository.save(freshProduct);
}
// JUNIOR NOTE: Optimistic locking is the recommended approach for most applications. It provides good performance without locking overhead. Use it when conflicts are rare (e.g., most web applications). The @Version annotation is all you need — Hibernate handles the rest.
Q82
What is pessimistic locking? How is it implemented?

Short answer

Pessimistic locking assumes conflicts are likely. It acquires a database lock on the record, preventing other transactions from modifying it until the lock is released. Implemented using SELECT ... FOR UPDATE.

In depth

// Pessimistic locking in JPA
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT p FROM Product p WHERE p.id = :id")
Product findByIdForUpdate(@Param("id") Long id);

// Repository method with explicit lock
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT p FROM Product p WHERE p.id = :id")
Optional<Product> findByIdWithLock(@Param("id") Long id);

// Using in transaction
@Transactional
public void updateProduct(Long id, Integer quantity) {
    Product product = productRepository.findByIdWithLock(id).orElseThrow();
    // Lock is acquired on database row
    product.setQuantity(quantity);
    // Lock is released at transaction commit
}

Lock mode types:

Lock ModeDescriptionWhen to use
PESSIMISTIC_READShared lock (others can read, not write)When reading, prevent concurrent updates
PESSIMISTIC_WRITEExclusive lock (others cannot read or write)When updating, prevent concurrent access
PESSIMISTIC_FORCE_INCREMENTLike WRITE + version incrementWhen you want to ensure version changes

Database-specific:

  • MySQL/PostgreSQL: SELECT ... FOR UPDATE
  • Oracle: SELECT ... FOR UPDATE
  • SQL Server: SELECT ... WITH (UPDLOCK)
// JUNIOR NOTE: Pessimistic locking is powerful but can cause deadlocks and performance issues. Use it only when you have high contention or need guaranteed consistency. Always keep the lock duration short and use in transactions.
Q83
Which locking type to prefer and when?

Short answer

Optimistic locking is preferred for most applications (web, REST APIs) because it's performant and scalable. Use pessimistic locking only when conflicts are frequent or when you need guaranteed exclusive access.

In depth

ScenarioRecommendedReason
Web applications with low conflict✅ OptimisticNo locking overhead, scales well
Banking/Financial transactions⚠️ PessimisticGuaranteed consistency needed
Inventory management✅ OptimisticConflicts rare, retry on failure
Counters with high contention⚠️ Pessimistic or AtomicOptimistic would retry too often
Batch processing⚠️ PessimisticLong-running, need data consistency
REST APIs (most cases)✅ OptimisticStateless, scalable, easy to retry
 
// Optimistic - for most cases
try {
    userRepository.save(user);  // Retry if OptimisticLockException
} catch (OptimisticLockException e) {
    // Retry logic
}

// Pessimistic - only when needed
@Transactional
public void transferMoney(Long fromId, Long toId, BigDecimal amount) {
    Account from = accountRepository.findByIdWithLock(fromId);
    Account to = accountRepository.findByIdWithLock(toId);
    from.setBalance(from.getBalance().subtract(amount));
    to.setBalance(to.getBalance().add(amount));
    // Locks released on transaction commit
}

Decision guide:

  • High conflict rate: Pessimistic locking (retry overhead is worse)
  • Low conflict rate: Optimistic locking (fast path)
  • Long transactions: Optimistic locking (pessimistic holds locks too long)
  • Short transactions: Pessimistic locking (acceptable)
  • External systems: Optimistic locking (less risk of deadlocks)
// JUNIOR NOTE: Start with optimistic locking for 90% of cases. Only move to pessimistic locking when you have evidence of high contention. Pessimistic locking can cause deadlocks and performance issues that are harder to debug than optimistic lock failures.
Q84
What is a DB index?

Short answer

A database index is a data structure that improves the speed of data retrieval operations on a table. It's like an index in a book — it allows the database to find rows without scanning the entire table.

In depth

Index types:

  • B-tree index: Default for most databases, good for equality and range queries
  • Hash index: Only equality queries, faster than B-tree for equality
  • Bitmap index: For low-cardinality columns (e.g., status, gender)
  • Full-text index: For text search
  • Unique index: Ensures uniqueness, used for primary keys
  • Composite index: Multiple columns, order matters
// Creating indexes in SQL
-- B-tree index (default)
CREATE INDEX idx_user_email ON users(email);

-- Unique index
CREATE UNIQUE INDEX idx_user_username ON users(username);

-- Composite index (order matters!)
CREATE INDEX idx_order_user_date ON orders(user_id, created_date);

// JPA index annotation
@Entity
@Table(indexes = {
    @Index(name = "idx_user_email", columnList = "email"),
    @Index(name = "idx_user_username", columnList = "username", unique = true),
    @Index(name = "idx_order_user_date", columnList = "user_id, created_date")
})
public class User {
    @Id
    @GeneratedValue
    private Long id;
    
    @Column(unique = true)  // Automatically creates unique index
    private String email;
}
// JUNIOR NOTE: Indexes are essential for performance but have costs. Every index adds overhead to INSERT, UPDATE, and DELETE operations. In JPA, @Column(unique=true) automatically creates a unique index.
Q85
Purpose of the index. Pros and cons. Why not create index for all possible combinations?

Short answer

Purpose: Speed up queries by reducing the amount of data scanned. Pros: Faster SELECT queries, unique constraints. Cons: Slower INSERT/UPDATE/DELETE, extra storage, maintenance overhead. You can't index all combinations because it would kill write performance and use too much storage.

In depth

Pros of indexes:

  • ✅ Faster SELECT: Sub-second lookups on large tables
  • ✅ Unique constraints: Enforces uniqueness
  • ✅ ORDER BY optimization: Indexed columns sort faster
  • ✅ JOIN optimization: Foreign key indexes speed up joins
  • ✅ Covering indexes: Query can be satisfied from index alone

Cons of indexes:

  • ❌ Write overhead: Every INSERT/UPDATE/DELETE must update all indexes
  • ❌ Storage space: Indexes can be as large as the data itself
  • ❌ Maintenance: Need to rebuild/update indexes periodically
  • ❌ Complexity: Wrong index can hurt performance
  • ❌ Query optimizer: May choose wrong index

Why not index all combinations:

For a table with 10 columns, possible combinations:
- Single columns: 10
- 2-column combos: 45
- 3-column combos: 120
- 4-column combos: 210
- ...and so on
Total: 2^10 - 1 = 1023 possible indexes

Real example: 50 columns → 2^50 - 1 possible indexes (impossible!)

Costs of all combinations:
1. INSERT/UPDATE/DELETE: Each operation updates 1000+ indexes → SLOW
2. Storage: Indexes = 10x-100x the data size
3. Memory: Indexes in cache = less space for data
4. Maintenance: Rebuilding indexes takes hours

Best practices:

  • ✅ Index columns used in WHERE, JOIN, ORDER BY
  • ✅ Index foreign keys
  • ✅ Create composite indexes for multiple conditions
  • ❌ Don't index low-cardinality columns (gender, status)
  • ❌ Don't index columns rarely used in queries
  • ❌ Don't create duplicate indexes
// JUNIOR NOTE: Indexes are a trade-off. Each index improves reads but hurts writes. Always ask: "Is this query run frequently enough to justify the cost?" Usually, 5-10 indexes per table is the sweet spot. More than that and you're probably over-indexing.
Q86
Explain briefly how a B-tree index works

Short answer

A B-tree is a balanced tree data structure where each node contains multiple keys. It allows logarithmic search time and is optimized for disk access. It supports both equality and range queries efficiently.

In depth

B-tree structure (simplified):

                    [30 | 60]
                   /    |    \
                  /     |     \
          [10 | 20] [40 | 50] [70 | 80 | 90]
          /   |   \   /   |   \   /   |   |   \
        ...  ...  ... ...  ...  ... ...  ...  ...

Key properties:
1. Balanced: All leaves are at the same depth
2. Each node: Contains multiple keys (K) and K+1 child pointers
3. Keys are sorted: Left < Parent < Right
4. Balanced automatically (splits/merges nodes)
5. Typical node size = database page size (4KB-16KB)

How lookup works:

Search for value 45:
1. Start at root: [30, 60]
2. 30 < 45 < 60 → go to middle child
3. Node: [40, 50]
4. 40 < 45 < 50 → go to middle child
5. Leaf node found → contains 45
6. Follow pointer to data row

Complexity: O(log n) disk I/O
For a B-tree with 500 nodes, only ~3 disk reads

Why B-tree for databases:

  • Low height: Even for millions of rows, height is small (3-5 levels)
  • Block-oriented: Each node = one disk block → efficient I/O
  • Supports range queries: <=, >, BETWEEN (unlike hash indexes)
  • Good cache usage: Nodes fit in cache/pages
  • Balanced: Predictable performance

B-tree vs B+tree:

  • B-tree: Data in both internal and leaf nodes
  • B+tree: Data only in leaf nodes (used by PostgreSQL, MySQL InnoDB)
  • B+tree advantage: More keys per node → wider tree → shallower
// JUNIOR NOTE: You don't need to implement a B-tree, but understanding how it works shows you understand database internals. Key takeaways: balanced, logarithmic search, range queries, and disk-optimized. This is why databases use B-trees for indexes.
Q87
Why not index all field combinations?

Short answer

Indexing all combinations leads to exponential growth in indexes, causing massive storage consumption and catastrophic write performance. The cost of maintaining indexes outweighs any read benefit.

In depth

The math:

For n columns:
- Single column indexes: n
- Composite indexes: 2^n - 1

For 10 columns: 1,023 indexes
For 20 columns: 1,048,575 indexes
For 50 columns: 1,125,899,906,842,623 indexes (impossible!)

Real costs:

CostImpact
StorageEach index can be as large as the table itself → 50 indexes = 50x storage
Write performanceEach INSERT/UPDATE must update every index → 10x-100x slower
MemoryIndexes in buffer pool → less space for data → more disk I/O
MaintenanceIndex rebuilds, statistics updates take hours → downtime
Query optimizerToo many choices → can choose wrong index → slower queries

Practical limits:

  • Typical tables: 5-15 indexes total (single + composite)
  • Heavy writes: Fewer indexes (3-5)
  • Read-heavy: More indexes (10-20)
  • Composite indexes: 3-4 columns max (beyond that, gains diminish)

Smart indexing strategy:

  • ✅ Index columns used in WHERE, JOIN, ORDER BY
  • ✅ Index foreign keys
  • ✅ Create composite indexes for common query combinations
  • ✅ Covering indexes for frequently executed queries
  • ❌ Don't index columns with low cardinality (boolean, enum)
  • ❌ Don't index columns you never query on
  • ❌ Don't index very large text fields (use full-text search)
// JUNIOR NOTE: Think of indexes as a budget. Each index costs storage and write performance. You have limited budget, so spend it on the queries that matter most. Measure query performance, identify bottlenecks, and index strategically. Don't just add indexes "just in case."
📋 Quick-reference cheat sheet
Fetch Types:
  EAGER → load immediately (default for ManyToOne, OneToOne)
  LAZY  → load on-demand (default for OneToMany, ManyToMany)
  ⚠️ Always prefer LAZY, use JOIN FETCH when needed

Relationships:
  @OneToOne   → 1:1, foreign key on owning side
  @OneToMany  → 1:N, inverse of @ManyToOne
  @ManyToOne  → N:1, owning side (has foreign key)
  @ManyToMany → N:N, join table

N+1 Problem:
  1 query for list + N queries for each item
  Fix: JOIN FETCH, EntityGraph, or EAGER (avoid)

Optimistic Locking:
  Uses @Version field (number or timestamp)
  Checks version on update → OptimisticLockException on conflict
  ✅ Preferred for most apps (performant, scalable)

Pessimistic Locking:
  Uses SELECT FOR UPDATE (database lock)
  Blocks other transactions
  ⚠️ Use only for high contention, short transactions

Indexes:
  Purpose: Speed up reads
  Pros: Faster queries, unique constraints
  Cons: Slower writes, storage, maintenance
  Best: Index WHERE, JOIN, ORDER BY columns
  ❌ Don't index all combinations (exponential growth!)

B-tree:
  Balanced tree with multiple keys per node
  O(log n) search, supports range queries
  Optimized for disk I/O
